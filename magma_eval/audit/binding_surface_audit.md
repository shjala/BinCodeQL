# Downstream binding surface audit: libxml2 regexp/automata module

**Date:** 2026-05-04
**Question:** Do any major language bindings of libxml2 expose the regexp/automata module
(`xmlAutomataNewNegTrans` and the four sister functions in `xmlregexp.c`) in their
public API surface, in a way that would let attacker-controlled input reach the
function without deliberate developer intent?

**Method:** GitHub code search via `gh api search/code` against each binding's repo,
querying for `xmlAutomataNewNegTrans`, `xmlNewAutomata`, `xmlAutomataNewState`, and
`xmlAutomataNewTransition`. For repos with hits, manual inspection of the matching
file's role (FFI declaration vs. application code).

## Major DOM-style bindings — no callers found

| Binding | Repo | Callers |
|---|---|---|
| PHP DOMDocument/SimpleXML | `php/php-src` | 0 application callers. The 3 hits in `ext/libxml/php_libxml2.def` are Windows DLL symbol-export entries, not application code. |
| Python lxml | `lxml/lxml` | 0 |
| Ruby Nokogiri | `sparklemotion/nokogiri` | 0 |
| Perl XML::LibXML | `shlomif/perl-XML-LibXML` | 0 |
| Swift FoundationXML | `swiftlang/swift-corelibs-foundation` | 0 |
| libxml2's own Python binding | `GNOME/libxml2/python/` | 0 |

These are the bindings whose public DOM-style APIs (DOM construction, XPath, XML
parsing, schema validation) are reachable from web/network input via normal
application code. None of them call into the regexp/automata module.

## SWIG-style FFI bindings — exposed but not used by high-level API

| Binding | File | Status |
|---|---|---|
| Node.js libxmljs (1060★, active 2026-01-12) | `lib/bindings/functions.ts` | Auto-generated by `swig2ts.js`. Lists the full libxml2 symbol table including all 5 automata functions of interest. **No high-level libxmljs API (parseXml, parseHtml, document.find, etc.) calls into the automata module.** Reachable only by `import { xmlAutomataNewNegTrans } from 'libxmljs/dist/lib/bindings/functions'` — a deliberate low-level import. |
| Free Pascal | `packages/libxml/src/xmlautomata.inc` + `xml2dyn.pas` | Pascal-translated C header + dynamic loader. Same SWIG-style "expose everything" pattern. No FPC application code calls it. |
| Delphi-Libxml2 (small project, ak47morozov) | `libxml2.pas` | Same — Pascal binding of the C header. |

These bindings expose the function via low-level FFI but require the developer to
deliberately reach into the FFI module and pass user input. That's equivalent to a
C developer including `<libxml/xmlautomata.h>` and calling the function directly.
Same axis (b) deliberate-misuse profile as the parent CVE's PoC.

## Conclusion

The reachability profile of `xmlAutomataNewNegTrans` and its four sister functions
is **(b) direct C-API misuse**, available equivalently from any language whose
binding exposes the full C API surface. It is **not** at axis (c1) — *indirect
reach via a normal DOM-style high-level API* — for any binding checked.

The parent CVE-2025-6021 (`xmlBuildQName`) earns its CVSS:N rating at (c1) because
PHP's `DOMDocument::createElementNS`, Swift's `XMLElement` constructors, etc. call
`xmlBuildQName` as part of routine DOM construction. None of those wrappers exist
for the regexp module.

## Methodology limitations

- GitHub code search has a 1000-result cap per query; broader queries
  (e.g. plain `xmlAutomataNewNegTrans` with no repo filter) returned 788 total
  hits, dominated by libxml2 mirrors, vendored copies (WebKit, FFmpeg builds),
  SDK dumps (Apple, iOS), and distro recipes.
- We did not check non-GitHub-hosted bindings (Debian, CPAN, RubyGems, etc. may
  carry private mirrors with patches not on GitHub).
- We did not check whether any consumer's *consumer* (i.e. a third-tier wrapper
  built on top of, say, libxmljs's low-level FFI surface) effectively wraps the
  function and exposes it through a higher-level API. Such a consumer would
  still face the same hardening burden.
- We did not check what happens at the binary symbol level when libxml2 is
  statically linked into a larger binary that exposes all symbols (e.g. via a
  Python `ctypes` user importing them dynamically). This is the same axis (b)
  deliberate-misuse path.
